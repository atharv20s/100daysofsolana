Explore Different Wallet Types

Comparing security, custody models, and convenience across Solana's wallet ecosystems (Day 5)

⌨️ CLI Keypair Wallet
Hot / Non-Custodial
  • Storage Location ~/.config/solana/id.json
  • Security Model Plaintext JSON File
  • Recovery Method File Backup / Seed Phrase
  • Setup Speed Instant (CLI command)
  • Safety Level Low (Hot & Plaintext)
Extremely scriptable and perfect for developers conducting local testing, scripting, or validator administration. However, it lacks confirmation prompts and password encryption at rest, making it insecure for staging mainnet funds.
🌐 Browser Extension
Hot / Non-Custodial
  • Storage Location Browser LocalStorage (Encrypted)
  • Security Model Password Lock + Confirmation Popups
  • Recovery Method 12/24 Word Seed Phrase
  • Setup Speed Fast (~2 Minutes)
  • Safety Level Medium (User Controlled)
The everyday gateway to Web3 dApps. Uses Wallet Standard to expose secure transaction signing interfaces. The password encrypts private keys at rest, and confirmation popups ensure no transaction signs silently.
📱 Mobile App Wallet
Hot / Non-Custodial
  • Storage Location iOS/Android Secure Enclave/Keystore
  • Security Model Biometric Authentication + Sandbox
  • Recovery Method 12/24 Word Seed Phrase
  • Setup Speed Medium (Download & Setup)
  • Safety Level Medium-High (Hardware-Backed)
Combines convenience with high-end OS sandboxing and biometric integrations (Face ID/fingerprint). Many modern smartphones leverage dedicated security co-processors to isolate key material from the standard system memory.
🛡️ Hardware Cold Wallet
Cold / Non-Custodial
  • Storage Location Offline Hardware Device (Ledger)
  • Security Model Physical PIN + On-Device Signatures
  • Recovery Method 24 Word Seed Phrase (Offline)
  • Setup Speed Slow (Physical configuration)
  • Safety Level Very High (Air-gapped)
The gold standard for asset protection. Private keys are generated offline and never exposed to the host computer's operating system. Transactions are passed to the device, signed in isolated hardware, and returned.
🤝 Multisig Smart Wallet
Programmable / Non-Custodial
  • Storage Location On-Chain Smart Contract State
  • Security Model M-of-N Signature Requirement
  • Recovery Method Member Key Replacement Program
  • Setup Speed Slow (Multi-party setup)
  • Safety Level Maximum (No single point of failure)
An on-chain account governed by smart contract rules (e.g., Squads). Requires multiple distinct private keys to sign and execute a transaction. Ideal for corporate treasuries, DAOs, or joint venture accounts.

1. Which wallet was fastest to set up?
The CLI wallet was the fastest to generate using `solana-keygen new`. It takes less than 5 seconds and instantly writes a key file without requesting password confirmations or completing seed-phrase verification tests.
2. Which felt safest?
The mobile wallet felt the safest due to its biometrics integration (Face ID/fingerprint) which provides hardware-backed gatekeeping. Additionally, the sandbox nature of mobile OS architectures restricts apps from snooping into the storage memory of other apps, reducing vulnerability to local malware.
3. Where is the private key stored in each case? Could you point to the exact file or storage location?
  • CLI Wallet: Stored on the local hard drive as a raw plaintext byte array in a JSON file, typically located at ~/.config/solana/id.json.
  • Browser Wallet: Stored in the browser's LocalStorage, but encrypted using the password created during setup. In Chrome on Windows, this is found in the local AppData directory under Chrome profile files.
  • Mobile Wallet: Stored in the phone's hardware-isolated key manager (iOS Keychain / Android Keystore), which is often backed by dedicated coprocessors (Secure Enclave).
4. If your laptop caught fire right now, which wallets could you recover? How?
Only the Browser wallet and Mobile wallet could be recovered, provided the 12/24 word recovery seed phrase was written down on paper or securely backed up offline. By entering this seed phrase into a new wallet extension or app, we derive the exact same public-private keypairs. The **CLI wallet** would be lost forever if the machine burned unless the backup id.json file or seed phrase was saved on external storage.
5. If you were building a dApp and needed to sign 500 test transactions in a script, which wallet would you use?
The CLI wallet. Because it is stored as a raw file, a Node.js or Python script can load it in memory and programmatically sign 500 transactions within seconds. Using a browser or mobile wallet would require manually clicking "Approve" on a popup 500 times, which is highly impractical.
6. If you were holding $10,000 in SOL, which wallet would you use?
A Hardware wallet (e.g. Ledger), ideally structured as one of the signer keys in a Multisig (e.g. Squads). This guarantees that private keys never interact with an internet-connected operating system, and requiring multiple signatures prevents a single exploit from draining the treasury.

Day 5 Key Takeaways

Our hands-on exploration of Solana's wallet ecosystem reveals that wallets are not accounts themselves; they are management interfaces for private keys. Under the hood, they all deal with the same mathematical concept—a public key (identity) and a private key (signing authority). The differences lie in where that private key is stored and how it gets utilized to sign transactions.

The tradeoffs are divided along two distinct dimensions:

  • Convenience vs. Security: A CLI wallet stores the key as a plaintext file. This makes it instantly scriptable but exposes it to simple system scripts. A hardware wallet isolates the key completely from the network, making it highly secure but tedious to sign transactions.
  • Custodian vs. Non-Custodial: Non-custodial wallets put the responsibility of backup (via seed phrases) entirely on the user. Losing the seed phrase is equivalent to losing the wallet.

Conclusion: In a real development workflow, you never rely on just one wallet type. Developers use CLI wallets for local scripts and continuous integration, browser wallets to interact with Web3 frontends and dev environments, and hardware cold wallets or multisigs to safeguard production funds and administrative keys.
